# 关于触发线生成和触发时机精确计算方式的说明

本程序中，需要精确的估计移动设备的位置，并且需要在刚好位于触发线上的时刻，触发一些操作的执行，这个精度需要控制在厘米级左右。

## 触发线生成
- 定义触发线为平行于坐标系X轴的一系列直线线段，线与线之间的间隔用参数指定，可以是恒定间隔，也可以是变化间隔。
- 开始触发位置和结束触发位置，分别定义了第一条和最后一条触发线的位置，该位置对应Y轴的数值。
- 目前，定义小区长度+过道宽度=触发线之间的间隔。
- 开始触发位置一定有第一条触发线，但是结束触发位置不一定刚好有触发线，只是代表超过这个位置不再进行触发逻辑的判断。
- 目前，点击 generateConfigButton 按钮之后，会生成小区并绘制在坐标系中。现在需要修改这个功能，点击 generateConfigButton 按钮之后，会生成触发线并绘制在坐标系中。触发线的长度为区域的宽度，即X轴的长度。触发线为一系列平行于X轴的直线线段。

## 触发时机计算方法
目前，我们通过高频的GPS信号（>= 10Hz ）来估计设备的位置，这个估计的精度在0.1-1米左右，但是这个精度在实际应用中是不够的，因为触发线上的位置需要精确到厘米级。因此，需要通过 Statestimator 实现精确的触发时机计算。StateEstimator::checkCrossing() 目前实现了是否穿越触发线的判断逻辑，现在需要更加精确的计算触发时机，即什么时刻设备刚好在触发线上。需要优化为以下算法：
- 通过类似窗口函数的方式，动态维护一个最近20个GPS点的平均速度大小v、平均速度方向v_theta、以及点之间的平均间隔d
- 判断触发时机时，按照粗估计和细估计两种方式：
    - 粗估计为每秒钟估计一次距离下一条触发线的距离，当这个距离小于等于 平均速度v * 3 时，进入细估计方式，即大约3秒内将越过触发线
    - 细估计方式为每接收一个GPS点，计算当前位置与触发线的距离dn，并维护一个倒计时定时器，定时器的值为dn / v，这个定时器相当于估计按照当前的速度v，需要dn / v秒才能到达触发线
    - 在细估计过程中，每次接收到一个有效的GPS点，都需要刷新这个倒计时定时器的值，即n=1,2,3，。。。n，在细估计的这n个点上，每次接收到一个点，都需要刷新这个倒计时定时器的值dn / v
    - 当倒计时定时器的值小于等于0时，说明设备刚好到达触发线，需要立即触发一些操作的执行
    - 以上设计，可以避免GPS点与点之间，因为离散性导致的触发时机误差
    - 同时，通过窗口函数的机制，维护设备最近20个GPS点的平均速度大小v、平均速度方向v_theta、以及点之间的平均间隔d，这样可以一定程度上避免因为GPS信号的不稳定性导致的计算误差放大效应

以上计算都是将经纬度映射到本地OA坐标系中的计算，距离单位是米，保留小数点后4位，即0.1毫米，时间单位是秒，保留小数点后4位，即0.1毫秒。倒计时定时器需要用时间戳模式，单位是微秒。以上单位换算时，需要注意保持高的计算精度。

## 触发时机的可视化
- 在运行以上算法时，需要在地图上可视化触发时机，即在地图上绘制触发时刻估计得出的触发位置
- 由于我们通过一个倒计时定时器来估计触发的时刻，所以在触发的瞬间，位置信息也需要进行估计，我们这里采用的方法是插值估计法，由于我们已知在触发时机到来之前最后一次卫星定位信息，而且我们也可知速度值和速度方向，那么我们就可以通过插值估计法，估计出在触发时机瞬间的位置。
