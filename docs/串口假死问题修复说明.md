# 串口假死问题修复说明

## 问题描述

程序运行后，GPS串口连接正常并能解析NMEA数据，但运行一段时间后会出现假死现象：
- 串口数据解析停止
- 程序本身未卡死
- 关闭串口再重新打开可恢复正常
- 问题反复出现

## 根本原因

### 1. 阻塞式串口读取导致线程卡死

**问题代码位置**：`src/core/fileio/serialport.cpp:102-137`

原代码使用了 `waitForReadyRead()` 进行阻塞式等待：

```cpp
// ❌ 错误的阻塞式读取
if (serial_port_->bytesAvailable() == 0) {
    if (!serial_port_->waitForReadyRead(100)) {  // 阻塞100ms
        return 0;
    }
}

while (bytes_read < size && serial_port_->waitForReadyRead(10)) {  // 继续阻塞
    // ...
}
```

**问题分析**：
- `waitForReadyRead()` 会让工作线程挂起等待数据
- 如果串口数据流中断或延迟，线程会一直阻塞
- 阻塞期间，Qt事件循环无法运行，导致：
  - 看门狗定时器无法触发
  - 信号槽无法处理
  - 线程看起来"假死"

### 2. Qt串口的正确使用方式

Qt的 `QSerialPort` 是**事件驱动**的，应该使用非阻塞式读取，而不是阻塞等待。

## 修复方案

### 修改1：非阻塞式串口读取

**文件**：`src/core/fileio/serialport.cpp`

```cpp
// ✅ 正确的非阻塞式读取
int SerialPort::read(char* buffer, int size) {
    if (!isOpen()) {
        return -1;
    }

    // 非阻塞式读取：只读取当前可用的数据
    qint64 available = serial_port_->bytesAvailable();

    // 如果没有数据可读，直接返回0（不阻塞等待）
    if (available == 0) {
        return 0;
    }

    // 读取可用数据（不超过缓冲区大小）
    qint64 bytes_to_read = qMin(static_cast<qint64>(size), available);
    qint64 bytes_read = serial_port_->read(buffer, bytes_to_read);

    if (bytes_read < 0) {
        qDebug() << "SerialPort: Read error" << serial_port_->errorString();
        return -1;
    }

    return static_cast<int>(bytes_read);
}
```

**关键改进**：
- 移除所有 `waitForReadyRead()` 阻塞调用
- 只读取当前可用的数据
- 没有数据时立即返回，不等待

### 修改2：优化数据采集线程休眠策略

**文件**：`src/core/sensor/sensorinterface.cpp`

**IMU线程**（第402-426行）：
```cpp
// 如果 IMU 数据有效，实时发送
if (imuData.time > 0) {
    // ... 处理数据 ...
    emit imuDataReceived(imuData);

    // 有数据时短暂休眠
    QThread::msleep(1);
} else {
    // 没有数据时休眠更长时间，避免空转
    QThread::msleep(10);
}
```

**GPS线程**（第751-762行）：
```cpp
// 如果数据有效，实时发送
if (gnssData.isvalid) {
    emit gnssDataReceived(gnssData);
    // 有效数据时短暂休眠
    QThread::msleep(5);
} else {
    // 无效数据时休眠更长时间
    QThread::msleep(20);
}
```

**优势**：
- 有数据时快速处理，减少延迟
- 无数据时降低CPU占用
- 避免过度轮询

### 修改3：添加缓冲区溢出保护

**新增方法**：
- `SerialPort::clearBuffer()` - 清空串口接收缓冲区
- `GnssSerialLoader::clearBuffer()` - 清空GNSS内部缓冲区
- `ImuSerialLoader::clearBuffer()` - 清空IMU内部缓冲区（已存在，增强）

**看门狗增强**（`sensorinterface.cpp`）：

```cpp
void GpsSensor::onWatchdogTimeout() {
    if (gnssLoader_ && gnssLoader_->isOpen()) {
        qint64 available = gnssLoader_->bytesAvailable();
        if (available > 0) {
            LOG_WARN("GPS watchdog triggered, %lld bytes available", available);

            // 如果缓冲区积压过多数据（超过2KB），清空缓冲区防止假死
            if (available > 2048) {
                LOG_WARN("GPS buffer overflow detected, clearing buffer");
                gnssLoader_->clearBuffer();
            }
        }
    }
}
```

**保护机制**：
- 看门狗每100ms检查一次缓冲区
- 如果缓冲区积压超过2KB，自动清空
- 防止旧数据堆积导致解析延迟

### 修改4：调整缓冲区大小

**文件**：`src/core/fileio/serialport.cpp:68-70`

```cpp
// 设置读取缓冲区大小为 8KB（原来是16KB）
serial_port_->setReadBufferSize(8 * 1024);
```

**原因**：
- 缓冲区过大会导致数据积压
- 8KB足够容纳短时间的数据突发
- 配合看门狗清理机制，保持数据新鲜度

## 技术要点总结

### Qt串口编程最佳实践

1. **永远不要在工作线程中使用阻塞式读取**
   - 避免 `waitForReadyRead()`
   - 避免 `waitForBytesWritten()` 的长时间等待

2. **使用非阻塞读取 + 轮询**
   - 检查 `bytesAvailable()`
   - 只读取当前可用数据
   - 配合适当的休眠时间

3. **缓冲区管理**
   - 设置合理的缓冲区大小
   - 定期检查缓冲区状态
   - 必要时清空旧数据

4. **线程安全**
   - 串口对象应该在工作线程中创建和使用
   - 使用信号槽跨线程通信
   - 避免跨线程直接访问串口对象

## 与Windows原生API的对比

原来使用Windows原生API（`CreateFile`, `ReadFile`等）没有问题，是因为：
- 原生API默认是非阻塞或可配置超时
- 更底层的控制权
- 不依赖Qt事件循环

切换到Qt后出现问题的原因：
- 误用了Qt的阻塞式API
- Qt串口需要配合事件循环使用
- 阻塞调用会干扰Qt的事件处理机制

现在的修复方案保持了Qt的优势（跨平台、易用），同时避免了阻塞问题。
