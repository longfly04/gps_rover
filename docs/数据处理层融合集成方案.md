## 1. 项目概述

### 1.1 项目背景

GPS-Seeder项目是一个基于Qt框架的GPS导航监控系统，旨在实现实时导航监控、历史轨迹回放、地图可视化等功能，项目地址为 D:\GPS-Seeder\gps_rover。
KF-GINS项目是一个基于扩展卡尔曼滤波的GNSS/INS集成导航系统，提供了成熟的IMU和GNSS数据解析功能，项目地址 D:\GPS-Seeder\KF-GINS\src 。

本方案旨在将KF-GINS项目的底层数据解析功能与GPS-Seeder项目进行融合，充分利用已有的GNSS解析和IMU解析代码，实现GPS-Seeder项目中的串口数据解析任务，同时为后续的融合导航功能做准备。

### 1.2 核心功能需求

- 实时导航监控：显示GNSS和IMU融合后的位置、速度、姿态数据
- 历史轨迹回放：支持播放之前记录的导航轨迹
- 地图可视化：支持多种底图切换，实时显示设备位置和轨迹
- 数据源切换：支持定位+惯性融合导航和直接使用粗精度GPS数据两种模式
- 支持路线生成：需要在底图上，支持路线绘制或者生成等，按照参数在多边形区域内生成弓型覆盖路线
- 测量功能：需要支持在地图上绘制测量线、测量距离、测量面积等功能

### 1.3 技术术语解释

| 术语 | 解释 |
|------|------|
| GNSS | 全球导航卫星系统(Global Navigation Satellite System)，包括GPS、北斗、GLONASS等 |
| IMU | 惯性测量单元(Inertial Measurement Unit)，用于测量物体的加速度和角速度 |
| KF-GINS | 基于扩展卡尔曼滤波的GNSS/INS集成导航系统 |
| 卡尔曼滤波 | 一种递归估计算法，用于从一系列包含噪声的测量数据中估计系统状态 |
| 低通滤波 | 一种信号处理技术，用于过滤高频噪声，保留低频信号 |
| 数据融合 | 将来自多个传感器的数据结合起来，以获得更准确的状态估计 |
| 串口通信 | 一种计算机与外部设备之间的数据传输方式 |
| WGS84 | 世界大地测量系统1984，一种全球坐标系统 |
| GCJ02 | 火星坐标系，高德地图等国内地图使用的坐标系统 |

- **TimeSync类**：实现了IMU和GNSS数据的时间同步功能，确保融合导航时数据的时间一致性。

### 3.1.1 数据源选择机制

为了支持上层应用灵活选择数据源，设计了SensorDataManager类，负责管理和切换不同的数据源。

#### 3.1.1.1 数据源类型

| 数据源类型 | 描述 | 适用场景 |
|---------|------|----------|
| GNSS_DIRECT | 直接使用GNSS原始数据 | 仅需要位置信息，对精度要求不高的场景 |
| KF_FUSION | 使用KF-GINS融合导航数据 | 需要高精度定位和姿态信息的场景 |
| SIMULATION | 使用模拟数据 | 测试和开发场景使用模拟数据 |

#### 3.1.1.3 数据源切换流程

1. **数据源初始化**：系统启动时，DataSourceManager初始化并设置默认数据源
2. **数据源选择**：上层应用通过UI或API调用setCurrentDataSource()选择数据源
3. **数据源验证**：DataSourceManager验证所选数据源是否可用
4. **数据源切换**：如果数据源可用，执行切换操作并发出dataSourceChanged信号
5. **数据获取**：SensorInterface根据当前数据源类型获取相应的数据

#### 3.1.1.4 数据源管理

- **数据源状态管理**：跟踪各数据源的可用性状态
- **数据源切换**：支持手动切换不同的数据源类型
- **信号通知**：当数据源切换时，通过信号通知相关组件

### 3.1.3 多线程架构

项目采用单进程多线程架构，具体如下：

| 线程名称 | 主要职责 | 通信方式 |
|---------|---------|----------|
| 主线程 | UI界面和用户交互 | 信号槽机制 |
| 串口数据线程 | 串口数据读取和解析 | 信号槽机制 |
| 数据处理线程 | 传感器数据处理和融合计算 | 信号槽机制 |
| 数据库线程 | 轨迹数据存储和管理 | 信号槽机制 |

多线程架构的优势：
- **更简单的通信机制**：线程间通过信号槽或共享内存通信
- **更低的系统开销**：线程创建和管理的开销比进程小
- **更简单的部署**：只需要管理一个进程
- **更好的资源共享**：线程间可以直接共享内存和资源

多线程架构的劣势：
- **隔离性较差**：一个线程崩溃可能影响整个进程
- **资源竞争**：需要注意线程安全和资源同步

### 3.3 数据流程设计

#### 3.3.1 传感器数据流程

1. **数据采集**：SensorInterface通过ImuSerialLoader和GnssSerialLoader读取IMU和GNSS的原始数据
2. **数据解析**：使用KF-GINS的解析模块解析原始数据，得到结构化的IMU和GNSS数据
3. **数据转换**：将解析后的数据转换为SensorData格式
4. **数据处理**：SensorInterface对数据进行处理，如时间同步、数据过滤等
5. **数据传递**：SensorInterface通过信号槽机制将数据传递给StateEstimator
6. **状态估计**：StateEstimator使用卡尔曼滤波或低通滤波估计系统状态
7. **数据分发**：SensorInterface通过信号槽机制将估计后的状态数据分发给UI层

#### 3.3.2 融合导航数据流程

1. **数据输入**：SensorInterface接收IMU和GNSS的原始数据
2. **数据预处理**：对输入数据进行预处理，如时间同步、坐标系转换等
3. **融合计算**：调用KF-GINS融合引擎进行状态估计和融合计算
4. **数据输出**：获取融合导航结果，如位置、速度、姿态等
5. **数据处理**：SensorInterface对融合结果进行处理，如坐标系转换、数据过滤等
6. **状态估计**：StateEstimator使用融合结果更新状态估计
7. **数据分发**：SensorInterface通过信号槽机制将融合后的状态数据分发给UI层

- **注释**：为所有类、函数和重要变量添加注释，使用Doxygen风格的注释格式

### 10.3 常见问题解答

#### 10.3.1 传感器连接问题

**问题**：无法连接到IMU或GNSS传感器

**解决方案**：
- 检查串口参数是否正确（端口号、波特率等）
- 检查传感器是否正常供电
- 检查串口线缆是否连接正确
- 使用串口调试工具测试传感器是否能够正常通信

#### 10.3.2 数据融合问题

**问题**：融合后的位置数据不准确

**解决方案**：
- 检查IMU和GNSS数据是否同步
- 调整卡尔曼滤波或低通滤波的参数
- 确保传感器校准正确
- 检查KF-GINS融合引擎的配置参数

#### 10.3.3 地图显示问题

**问题**：地图不显示或位置显示不正确

**解决方案**：
- 检查网络连接是否正常
- 检查高德地图API密钥是否正确
- 检查坐标转换是否正确（WGS84 vs GCJ02）
- 检查WebEngineView是否正确初始化

#### 10.3.4 性能问题

**问题**：系统运行缓慢或卡顿

**解决方案**：
- 检查是否有内存泄漏
- 优化数据处理算法
- 调整线程优先级和调度策略
- 减少UI更新频率
- 优化地图显示和轨迹绘制
